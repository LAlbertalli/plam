# PLAM - Technical Debt & Future Improvements

This document tracks identified issues, technical debt, and architectural improvements that should be addressed in the future, but are not currently blocking active development.

## Security & Configuration
- [ ] **Database Credentials**: The PostgreSQL database configuration and credentials should not be hardcoded (e.g., in `config.py` or the `docker run` command). Ensure passwords are not the default "postgres" and use `.env` files or a secret manager.

## Reliability & Execution
- [ ] **Robust Container Health Checks**: The `DockerManager.start_db` method currently relies on a simple `time.sleep(3)` to wait for the container. It should be refactored to implement a proper polling loop that checks for database readiness (e.g., repeatedly trying to open a connection until successful). Furthermore, this "wait for readiness" strategy should be generalized for all container spin-ups (such as the `llama.cpp` model instances).
- [ ] **Database Volume**: The PostgreSQL container should have its data volume mounted to `plam/data/db` on the host to ensure data persistence across container restarts.
- [ ] **Active Model Scarcity Queue (On-Hold Requests)**: Implement a request queueing and scheduling system inside `OrchestratorService`. When a request is made for a model that is currently not available (e.g., because system RAM is exhausted and all other active models are locked and currently in use), the chat or autonomous agent request should be placed "on hold". The client should receive a status stream message (e.g., `"LLM Server busy. Request queued..."`), and the orchestrator should automatically wake up and resume the chat stream once an active stream terminates and frees up the required system RAM.

## Architectural Lifecycle
- [ ] **Rethink Manual Start/Stop**: Currently, models have manual Start/Stop endpoints exposed to the UI for testing. We should rethink this approach and consider fully automating the lifecycle (e.g., the Orchestrator Proxy automatically starts/stops models based on demand and LRU cache). Also, we should show transient states like 'spinning up', 'shutting down', etc. for the models in the UI.
- [ ] **Stateful Active Model Cache & Stateless ProxyService**: Reconsider moving from the current stateless singleton `ProxyService` architecture (where database sessions and model IDs are passed on every call) to an in-memory cached state representation (`ActiveModel` registry wrapper). This will avoid querying database rules on every single message turn. See full design analysis at [active_model_caching_analysis.md](file:///home/luca/plam/doc/design/active_model_caching_analysis.md) for potential corner cases like cache invalidation on edits, multi-worker process memory isolation, and container health synchronization.
- [x] **Multi-Worker Shared Memory / Distributed State Tracking**: Transition the in-memory `_active_streams` registry inside `DockerManager` to a shared memory state tracker. This is critical for scaling FastAPI to multiple worker processes (e.g., `gunicorn` / `uvicorn` with `--workers N`) where isolated local python memories would cause active stream counters to become out of sync. *Done: Implemented standard process Shared Memory (`multiprocessing.shared_memory.SharedMemory`) synchronized via workspace file locking.*
- [ ] **LRU Eviction & Thrashing Cooldown**: Implement a Least Recently Used (LRU) model eviction policy within `DockerManager.start_model` to decide which model container to stop first when freeing up system memory. In addition, introduce a configurable cooldown period (e.g., `COOLDOWN_PERIOD_SECONDS` = 60) during which a newly started or recently utilized model container cannot be evicted, preventing rapid startup/shutdown container thrashing.
- [ ] **Server-Side Multi-Agent Delegation**: Reconsider passing target `agent_id` in message payload (`POST /sessions/{id}/message`) to eliminate client-side logic selection security risks and enable robust parallel subagent coordination. Transition sessions to bind a `primary_agent_id` upon creation and run all subagent triggers programmatically on the server. See full analysis at [multi_agent_delegation_architecture.md](file:///home/luca/plam/doc/design/multi_agent_delegation_architecture.md).

## Future Features & UI/UX Roadmap
- [ ] **Markdown Support in Chat**: Ability to parse and render Markdown format in model responses (both for the main assistant response and the collapsible thinking trace).
- [ ] **Dynamic Llama Request Options**: Ability to send custom options (like `temperature`, `top_p`, or reasoning-specific flags) in the LLM request to change model parameters. This requires saving model-specific options templates in the database and loading them dynamically in the Orchestrator.
- [ ] **Model Page Redesign**: Redesign the model list and management page (`/models`) to be less clunky, cleaner, and more user-friendly.
- [ ] **Chat Session Naming**: Add the ability to rename chat sessions directly from the chat UI (editing title).
